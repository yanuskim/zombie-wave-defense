# 좀비 웨이브 디펜스 — 완성도 진단 & 로드맵

작성일: 2026-06-11 · 대상: game.js(1,822줄), index.html, styles.css 전수 점검 결과

## 요약

게임의 뼈대(직업, 유물, 상점, 도감, 웨이브, 핫시트 팀플레이)는 잘 잡혀 있다. 하지만 현재 상태로는 **어떤 플레이를 해도 약 6일차에 무조건 죽음 엔딩**이 나오는 치명적 설계 버그가 있고, 진행 상황이 저장되지 않아 새로고침하면 모든 재화와 강화가 사라진다. 완성도를 올리려면 화려한 기능 추가보다 이 두 가지부터 고쳐야 한다.

---

## 1. 치명 버그 (최우선 수정)

### 1-1. 탈수/배고픔을 회복할 방법이 없음
- recordActionCost()(game.js:1303)에서 행동 1당 탈수 -3, 배고픔 -1로 깎이지만, **탈수/배고픔을 올리는 코드가 게임 전체에 존재하지 않는다.**
- 탈수 100 ÷ 3 = 약 33행동 = 약 5~6일차에 모든 게임이 죽음 엔딩으로 강제 종료된다. 식량/물을 아무리 모아도 소용없다.
- 수정안: "식사/식수" 행동 추가(식량 1 → 배고픔 +20, 물 1 → 탈수 +25 등) 또는 턴 종료 시 식량/물 소비가 게이지를 자동 회복하도록 연결.

### 1-2. 상호작용(interact) 버튼이 없음 — 죽은 코드
- interact() 함수(game.js:1516)는 완성돼 있지만 index.html 행동 버튼 7개에 data-action="interact"가 없고, 클릭 핸들러(game.js:1716)에도 분기가 없다.
- 나무/거미줄/돌/흙을 얻는 유일한 수단이라 **제작 시스템 대부분(창, 텃밭, 요리대 등 12개 레시피)이 사실상 도달 불가능**하다.
- 수정안: 행동 밴드에 "상호작용" 버튼 추가 + 핸들러 연결. (8차 브리프에 명시된 기능)

### 1-3. 기지 체력 상한 불일치
- 기지 최대치는 8인데, 기지/공사장 이벤트(game.js:72, 88)는 Math.min(100, base + 8~10)으로 8을 초과해서 올린다. "8/8" 표시와 모바일 게이지가 깨진다.
- 수정안: 상한을 8로 통일하거나, 기지 체력 자체를 100 스케일로 리디자인.

### 1-4. 게임 중 카드 강화 = 무한 회복 치트
- upgradeCard()(game.js:1274)가 플레이어를 createPlayer()로 재생성해서 **체력과 행동력이 풀로 리셋**된다. 보석 30개로 전체 회복+행동력 충전이 가능.
- 수정안: 강화 시 기존 hp/actions를 보존하고 보너스 수치만 반영.

### 1-5. 죽은 플레이어가 계속 행동 가능
- 개인 hp가 0 이하로 떨어져도 턴이 돌아오고 행동할 수 있다. 전멸일 때만 엔딩 체크. 음수 체력으로 계속 플레이된다.
- 수정안: hp ≤ 0이면 행동 불가(관전) 처리, 턴 순서에서 제외.

## 2. 게임플레이·밸런스

### 2-1. 전투가 의미 없음
- attack()은 데미지를 계산하지만 좀비 HP를 깎는 개념이 없고, 결과는 기지 +1 회복 정도. 밤 웨이브 피해는 공격 횟수와 무관하게 들어온다.
- 제안: 낮에 처치한 좀비 수만큼 밤 웨이브에서 실제로 제외되는 구조로 변경. "이번 밤에 올 좀비: 시민좀비 2, 육상좀비 1 (처치로 1 감소)" 식 표시. 전투에 목적이 생기고 행동력 배분 고민이 생긴다.

### 2-2. 엔딩 도달 동선 검증 필요
- 백신 엔딩: 연구 4회 = 데이터 8 + 의약품 4 필요 → 연구소 조사 8회 + 이동. 약 5일 소요로 탈수 데드라인과 거의 겹친다(1-1 수정 전엔 불가능).
- 최강의 기지 엔딩: 식량 24 + 물 24 + 무기 4. 턴마다 식량/물이 인원수만큼 소모돼 솔로 기준 약 10일 필요 → 현재 웨이브 압박(기지 8)으로는 사실상 불가능.
- 제안: 1-1 수정 후 시뮬레이션(스크립트)으로 각 엔딩 도달 가능 일수를 검증하고 수치 조정.

### 2-3. 난이도가 이지 하나뿐
- difficulty 셀렉트에 옵션이 easy 하나, easyWaves만 존재. 노말/하드 웨이브 테이블 추가가 자연스러운 다음 단계.

### 2-4. 제작이 자동 선택
- craft()가 재료 맞는 첫 레시피를 임의로 만든다. 플레이어가 레시피를 고를 수 없어 전략성이 없다.
- 제안: 제작 가이드 패널의 레시피 카드를 클릭하면 해당 아이템을 제작하도록 변경.

### 2-5. 직업 간 밸런스
- 사냥꾼(체력 250/공격 20)이 의사(체력 100)보다 모든 면에서 압도적. 의사의 치료는 1-1 수정 전엔 가치가 낮다. 시민은 고유 능력이 없음(물 +1은 숨겨진 효과).
- 제안: 시민에게 명시적 고유 능력 부여(예: 행동력 +1 또는 조사 보너스), 의사 치료를 부활/감염 치료 등 대체 불가 역할로.

## 3. UX·모바일

- **저장 없음**: 코인/보석/카드/강화/업적(meta)이 메모리에만 있어 새로고침하면 전부 소실. localStorage 저장/복원이 체감 완성도를 가장 크게 올리는 단일 작업.
- **토스트 스팸**: 모든 로그가 토스트로 떠서(game.js:1681) 연속 행동 시 메시지가 덮어써진다. 중요 이벤트만 토스트로, 나머지는 로그만.
- **행동 버튼에 비용 미표시**: 탐색 2, 치료 2, 탐험 4 등 비용이 버튼에 없어서 외워야 한다. 버튼에 비용 배지 추가.
- **재료 부족 사전 안내 없음**: 행동력을 쓰고 나서 "재료 부족"이 뜨는 경우가 있다(craft는 spend 후 실패 체크). 불가능한 행동은 버튼을 비활성화하거나 행동력을 환불.
- **모바일 미디어쿼리 4개 존재**: 기본 대응은 돼 있으나 iPhone 실기 테스트 체크리스트 필요(터치 타깃 44px, 안전영역, 가로모드).
- **설정/우편함 등이 설명 패널뿐**: 미구현 기능은 "준비 중" 배지로 명확히 구분하면 테스터 혼란이 줄어든다.

## 4. 비주얼·연출

- 보유한 캐릭터 이미지(char-*.png, face-*.png)가 카드 초상화에만 쓰인다. 토의 화면, 파티 리스트, 엔딩 화면에도 활용 여지가 큼.
- 밤 웨이브가 텍스트 로그 한 줄로 끝난다. 가장 중요한 순간이므로 **"밤 결과" 전용 오버레이**(좀비 등장 → 기지 피해 → 날짜 전환 연출)를 추천. 기존 base-hit 효과를 확장하면 된다.
- 엔딩 6종이 dialog 텍스트뿐. 엔딩별 일러스트/색상 테마만 넣어도 보상감이 크게 다르다.
- 배경음이 단일 드론 사운드. 버튼/공격/피해/엔딩 효과음 4~5개(Web Audio 합성으로 충분)만 넣어도 타격감이 산다.
- 장소 이동 시 맵 노드 간 이동 애니메이션(현재 위치 마커 이동)이 없어 위치 변화가 눈에 안 띈다.

## 5. 안정성·코드 구조

- render()가 매번 전체 화면을 다시 그리고 리스너를 재바인딩한다. 규모가 커지면 버그 온상이 되므로, 장기적으로 이벤트 위임(컨테이너에 리스너 1개) 방식으로 전환 권장.
- 상점 1초 인터벌(game.js:1815)이 상점 열려 있는 동안 매초 DOM을 재생성 — 구매 버튼 클릭이 씹힐 수 있다. 타이머 텍스트만 갱신하도록 분리.
- 데이터(직업/레시피/웨이브)와 로직, DOM이 한 파일에 섞여 있다. 당장은 유지해도 되지만, 밸런스 조정이 잦아지면 데이터를 별도 JS/JSON으로 분리하면 편하다.
- 랭킹은 가짜 데이터, 랭킹 업적은 test: () => false 고정 — 의도된 목업이면 OK, README에 명시 권장.

## 우선순위 로드맵

### Phase 1 — 게임이 "성립"하게 만들기 (치명 버그)
1. 식사/식수 회복 메커니즘 추가 (1-1)
2. 상호작용 버튼 연결 (1-2)
3. 기지 체력 상한 통일 (1-3)
4. 강화 회복 치트 수정 (1-4)
5. 사망 플레이어 처리 (1-5)
6. localStorage 저장/복원

### Phase 2 — "재밌게" 만들기 (밸런스·핵심 루프)
7. 공격이 밤 웨이브를 실제로 줄이도록 전투 개편 (2-1)
8. 엔딩 도달 시뮬레이션 + 수치 조정 (2-2)
9. 레시피 선택 제작 (2-4)
10. 직업 밸런스/시민 고유 능력 (2-5)

### Phase 3 — "보기 좋게" 만들기 (연출·UX)
11. 밤 결과 오버레이 연출
12. 행동 버튼 비용 표시 + 불가능 행동 비활성화
13. 엔딩별 비주얼 차별화
14. 효과음 4~5종
15. 토스트 정리

### Phase 4 — "넓히기" (콘텐츠)
16. 노말/하드 난이도 웨이브 테이블
17. 유물 추가(현재 2종) + 등급 확률 실제 적용
18. 퀘스트/출석 실동작
19. 코드 구조 정리(데이터 분리, 이벤트 위임)

각 Phase는 끝날 때마다 플레이 테스트 1회를 권장. Phase 1만 끝나도 "버그 있는 데모"에서 "플레이 가능한 게임"이 된다.
